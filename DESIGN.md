> 16 个月。41 倍。
>
> Anthropic 的数据显示，今天的 AI 完成同一个软件任务的速度，是 16 个月前的 41 倍。
>

>
> 我们正在进入一个执行力暴涨的时代。写代码、画图、出报告——这些"做出来"的门槛，正在以你我都没预料到的速度趋近于零。
>
> 但有一件事 AI 给不了你：**判断力**。
>

>
> 这才是每个人在 AI 时代真正的护城河。
>
> 这篇文档想讲的是：**如何把你已有的专业判断力，接上 AI 的执行力。**

### 第一节：你现在在用 AI 做什么

**定位**：拉近距离，让读者觉得"这说的是我"

**内容要点：**
- 你可能在用 ChatGPT 改文字、翻译、解释 error message，或者生图，再复制粘贴回去
- 这已经很有用了，但有个共同的操作模式：你贴内容 → 它回答 → 你复制 → 你再贴下一段
- 这种模式有个名字：你在替 AI 打工

**语气**：轻松，带一点自嘲式幽默，不批判

---

### 第二节：差在哪里

**定位**：明确痛点，为下一节铺垫

**内容要点：**
- 它不认识你的文件——你得把数据复制进对话框，超过几十行就开始出错
- 它不能真正跑代码——给你一段 Python，你还得自己去装环境、跑、再把报错贴回来
- 每次新对话记忆清零——你的项目背景、上次的决定、数据格式，都要重新解释
- 它不知道结果对不对——生成的图你还得眼睛盯着看，它自己不会检查

**核心比喻**：像一个记忆力为零、不会用电脑的高级顾问——你得把所有材料念给他听，他给意见，你回去自己执行，执行完你再来汇报

**语气**：让读者觉得"对！就是这个感觉！"——这不是在批评 AI，而是说"这不是 AI 的错，是使用方式的问题"

---

### 第三节：AI 终端智能体是什么

**定位**：核心概念，讲清楚不讲复杂

**内容要点：**
- "终端"不是重点，重点是：它能直接操作你电脑上的文件和程序
- 你说"帮我分析这份数据"——它会自己打开文件、写代码、运行、看结果、发现问题、修改、再运行
- 你不需要做中间人，它自己完成整个循环
- 你的角色从"执行者"变成"审批者"——它干活，你决定结果要不要

**比喻升级：**
- 之前：隔着玻璃的顾问，你念材料给他听，他出主意，你回去执行
- 现在：直接坐在你旁边的博后，你说目标，他自己开电脑、找文件、跑分析、给你看结果

**需要说清楚**：它会犯错，所以"审批者"这个角色不能省——这不是缺点，这是正常的工作流

### 第四节：工具选哪个

**定位**：给国内读者清晰的决策路径，用新闻故事建立信任

#### 工具推荐表

| 工具 | 定位 | 门槛 | 费用 |
|------|------|------|------|
| **Kimi Code** | 国产终端智能体，直接对标 Claude Code | 中等 | 按 token，约 Claude 1/10 |
| **Cursor** | 带 AI 的代码编辑器，界面友好 | 最低，像用 Word | 免费版可用 |
| **Claude Code + Kimi API** | 用 CC 框架，接 Kimi 模型 | 稍高 | 同 Kimi 价格 |

#### 推荐逻辑（在文档里明说）
- 第一次接触 → 先试 Cursor，有界面、门槛最低
- 想用终端智能体、国内网络友好 → Kimi Code
- 认真用起来做科研项目 → Claude Code + Kimi API

#### "Kimi 为什么靠谱"的故事

> **2026年3月，一件有意思的事被曝光**：Cursor（全球最受欢迎的 AI 编程工具之一）的新旗舰模型 Composer 2，被社区发现底层用的是 Moonshot AI（月之暗面）的 Kimi K2.5 模型。Cursor 联合创始人随后承认，"我们从 Kimi K2.5 的开源基础上继续训练"。Kimi 官方也回应：这是授权合作，"我们为 Kimi K2.5 成为 Composer 2 的基础感到骄傲。"

> 同一时期，另一家国内大模型公司 MiniMax 被指在自家产品中混入了 Kimi 的代码（连 kimi 字样都没改干净）——真实性存疑，但可以说明 Kimi 在代码能力上已经处于国内第一梯队。

#### 省钱操作提示
在 Claude Code 里改两行环境变量，就能把底层模型换成 Kimi——框架用 Claude Code，模型用 Kimi，成本约降至原来的 1/10。

```bash
export ANTHROPIC_BASE_URL="https://api.moonshot.cn/anthropic"
export ANTHROPIC_API_KEY="你的 Kimi API Key"
```

### 第五节：课题组能用来做什么

**定位**：最核心的一节，让读者看到跟自己直接相关的价值

每个场景配一个真实感的对话示例：

#### 场景一：批量处理数据文件

> 你有 30 个 Excel 表，每个是一个样品的测量结果，格式略有差异。
> 告诉 AI："帮我把这些文件合并，按样品编号整理，输出一张汇总表，标注哪些样品数据缺失。"
> 它会自己打开每个文件、处理格式不一致的问题、生成结果，中途发现异常还会告诉你。

#### 场景二：生成图表

> "用这份数据画箱线图，x 轴是处理组别，y 轴是细胞活力，颜色区分对照和实验组，存成 300dpi tif，顺便告诉我各组的 p 值。"
> 之前这件事要么你自己写 Python，要么在 Origin 里一步一步操作。现在说一句话，它跑完告诉你结果。

#### 场景三：读文献辅助方法设计

> 把一篇参考文献的方法部分发给它，问："他们的统计方法是什么？我现在有 3 组、每组 6 个样本，用他们的方法合适吗？有没有更好的选择？"
> 它不会替你思考科学问题，但它能帮你快速摸清文献里的技术选型，节省你查资料的时间。

#### 场景四：期刊投稿排版

> 不同期刊要求不一样：Nature 系列要双倍行距、参考文献 Vancouver 格式；某些中文核心要宋体五号、图题中英文双语；投 Cell 子刊又是另一套。
>
> 告诉 AI："帮我把这篇稿子改成《XXX》的投稿格式，参考他们官网的 Author Guidelines，图序从 Figure 改成 Fig.，参考文献格式统一。"
> 它会读你的文件、对照要求逐项修改，改完告诉你哪些地方需要你自己确认。
>
> 如果同一篇稿子被拒、要投另一个刊，再说一句："按《YYY》的格式重新排"——不用重新手工改一遍。

#### 场景五：接手"历史遗留代码"

> 前任学生离组，留下一堆没有注释的脚本，文件名叫 `analysis_final_v3_REAL_use_this.py`。你不确定它在算什么、数据从哪来、改哪里会出问题。
>
> 把整个文件夹交给 AI："读懂这些代码，告诉我数据流是什么、每个脚本在做什么、变量命名的规律是什么。"
> 它会给你一份白话说明，帮你搞清楚这段代码的逻辑，而不需要你自己一行一行去读。
>
> 然后你可以说："我现在换了新的仪器，输出格式变了，帮我把数据读取部分改一下，其他逻辑不动。"
> 它能精准定位需要改的地方，不会把整个脚本翻一遍。
>
> 组里换人、项目交接、导师让你复现两年前的结果——这些场景下，AI 等于帮你补了一个随时在线的"老人"。

**一句话总结**：这些任务的共同点——之前要么你学了语言才能做，要么反复手工操作。现在门槛变成：你知道自己想要什么结果就够了。

### 第六节：进一步扩展——Skill 与 MCP 生态

**定位**：让读者意识到工具可以深度定制成课题组专属助手

#### Skills：可复用的专家经验

Skill 是一段写好的指令集，告诉 AI 遇到某类任务该怎么做。

**重要的是**：这些 Skill 不需要你自己写。Claude Code 有一个开放社区，全球开发者和研究者持续贡献各类 Skill——统计分析、代码审查、文献阅读、论文润色……你可以直接安装使用。

课题组自己能写的：
- `/投稿格式` — 按期刊 Author Guidelines 排好稿子
- `/统计检验` — 根据实验设计推荐合适方法
- `/数据质检` — 自动检查缺失值、异常值、单位不一致
- `/摘要润色` — 按"方法-结果-结论"结构检查摘要，标出逻辑跳跃

#### MCP：连接你已经在用的工具

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，让 AI 能直接调用外部应用。同样有庞大的开源社区生态。

| MCP | 能做什么 |
|-----|---------|
| Notion / 飞书 | 实验记录自动整理进数据库，生成会议纪要 |
| GitHub | 读仓库、提交修改、管理版本 |
| 邮件客户端 | 起草审稿回复、投稿确认邮件 |
| 文献数据库 | 查 PubMed、导出引用格式（需额外配置）|

官方 MCP 列表：`modelcontextprotocol.io`

#### 为什么生态很重要

> 你现在不需要是专家，但你可以直接用专家写好的技能。
>
> 当一个统计学家把他的分析方法封装成 Skill，当一个资深编辑把润色逻辑写进工作流，这些经验就变成了你可以直接调用的能力——不需要你从头学。
>
> 先站在巨人的肩膀上把事情做出来。在这个过程里，你会慢慢理解背后的逻辑。
>
> 当你摸清了自己课题组的工作流——怎么清洗这类数据、投这个领域的期刊有哪些注意事项、你们组跑质谱的标准流程是什么——你就可以把这些经验写成 Skill，发出去，让全世界做同类研究的人都能用。
>
> **你不只是用工具的人，你也可以是给工具加能力的人。**

## 从 Anthropic 博客提炼的科研版建议

（可穿插到各节，或作为单独的"使用技巧"板块）

| 原则 | 科研版解读 |
|------|-----------|
| Demo 优先，而非文档优先 | 先让 AI 跑出一张图，再讨论方法对不对。与其写两页分析思路，不如五分钟出结果 |
| 短周期试验，而非大计划 | 不要想着"把整个分析流程 AI 化"，先找今天最烦的一件事试试 |
| 旧流程用新模型重跑 | 三个月前觉得 AI 做不到的事，现在可能可以了。定期把旧的难题再问一遍 |
| 做最简单的版本 | 先用最直白的语言说清楚你要什么，别花太多时间调 prompt |
| 模糊角色边界 | 你不需要成为程序员才能跑代码，不需要是统计学家才能做统计——你需要的是判断结果对不对的眼光 |
