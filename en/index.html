Practical Starter Guide

Using AI for Research: A Practical Guide for Non-CS Labs

Connect the professional judgment you already have to AI's execution power, without becoming a programmer.

16 months. 41x. Anthropic data: AI completes the same software task 41 times faster than it did 16 months ago.

Naval said: software was eaten by AI. Anthropic's head of product said: the path from idea to prototype has already collapsed from weeks to hours.

We are entering an era of exploding execution power. Writing code, making figures, producing reports - the barrier to "getting something built" is falling toward zero faster than most of us expected.

But there is one thing AI cannot give you: judgment.

Knowing what experiment to run, noticing what is wrong with a dataset, seeing where the logic breaks in a manuscript - that comes from the professional intuition you have accumulated in your field. Someone put it well: AI's capability boundary is your knowledge boundary, your command of field-specific vocabulary, and your curiosity about the details of the domain.

That is the real moat anyone has in the age of AI.

This document is trying to explain one thing: how to connect the professional judgment you already have to AI's execution power.

How You Currently Use AI

You probably use ChatGPT to rewrite text, translate, explain error messages, or generate images, then copy and paste the result back into your work. Paste something in, get an answer, copy it out, paste the next thing in.

That is already useful. But notice the operating pattern: you are doing all the moving. You read the file, you copy the content, you run the code, you bring back the output, and then you paste it in again.

There is a name for this pattern: you are working for the AI.

You have become the interface between the AI and your real work, manually transporting information back and forth over and over. The AI sits behind glass; you are the one pressing all the buttons.

What's Missing

The copy-paste loop exists because the way most people use AI today has four structural limits:

  • It does not know your files. You have to copy data into the chat box. Once you paste more than a few dozen rows, it starts making mistakes or simply ignores the end.
  • It cannot really run code. It writes Python for you, but you still have to install the environment, run it yourself, and paste the error back.
  • Its memory resets with every new chat. Your project background, previous decisions, and data formats all have to be explained again from scratch.
  • It does not know whether the result is correct. The figure gets generated, but the AI will not look at it and judge whether it makes sense. Only you can do that.

It is like a high-level consultant with zero memory who also cannot use a computer. You have to read all the materials out loud, they give advice, you go back and execute it yourself, and when you return to report what happened, they have already forgotten the last conversation.

This is not criticizing AI. It is describing a way of using it, and that way of using it can change.

What an AI Terminal Agent Is

An AI terminal agent removes the glass. Instead of waiting behind a chat window for you to paste things in, it directly operates files and programs on your computer: opening files, writing code, running it, checking the result, spotting problems, revising, and running again, all by itself.

You say, "Help me analyze this dataset," and it opens the file, writes the code, runs it, looks at the output, notices a problem, fixes it, runs it again, and then shows you the result. You did not have to do anything in between.

Your role shifts from executor to approver. It does the work; you decide whether the result is acceptable.

Before copy-paste loop

You open the file. You copy a piece of data. You paste it into the chat. You ask a question. You copy the code it returns. You open a terminal. You run the code. You get an error. You copy the error. You paste it back. You copy the fix. You run it again. You copy the output. You paste it back for interpretation. You copy the interpretation. You paste it into your notes.

Now terminal agent

You describe what you want. It opens the file, writes code, runs the code, reads the output, notices the error, fixes it, reruns it, and returns the result to you. You look once and decide whether to continue.

One point needs to be explicit: it will make mistakes. That is why the "approver" role cannot be removed. This is not a flaw in the workflow; it is the workflow itself. You provide the judgment, and it provides the execution.

Which Tool to Use

There are four tools worth knowing about, each with a different balance between zero setup and full capability.

Tool Positioning Barrier Cost
happycapy An AI-native computer in the browser, with no environment setup required Lowest, open and use it Free tier available, Max has unlimited tokens
Kimi Code A domestic terminal agent that directly competes with Claude Code Medium Token-based, about 1/10 the cost of Claude
Cursor An AI-enabled code editor with a friendly interface Low, like using Word Usable on the free tier

How to decide

  1. Zero barrier, start immediately -> happycapy: open it in the browser and start. No software installation, no environment setup, no command line. Process data, make figures, write reports, build pages, and get results by describing what you want.
  2. First time touching a terminal agent -> Try Cursor first. It has an interface, the barrier is low, and you immediately understand the "approver" model.
  3. Want a terminal agent with China-friendly network access -> Kimi Code, which is compatible with the Anthropic API and does not require a VPN.
  4. Want to seriously use this for research projects -> Claude Code + Kimi API. The ecosystem is the most mature, while the pricing stays at Kimi levels.

Why Kimi is credible

In March 2026, something interesting came to light: Composer 2, the new flagship model from Cursor, one of the world's most popular AI coding tools, was found by the community to be built on Moonshot AI's Kimi K2.5. Cursor's co-founder later acknowledged, "We trained on top of the Kimi K2.5 open-source base." Kimi also responded: "We're proud that Kimi K2.5 is the foundation for Composer 2."

Around the same time, another domestic model company, MiniMax, was accused of mixing Kimi code into its own product, even leaving visible "kimi" strings behind. The claim is difficult to verify, but it does suggest that Kimi is already in the first domestic tier for coding capability.

Cost-saving setup

Change two environment variables in Claude Code and you can switch the underlying model to Kimi. Keep the Claude Code framework, use the Kimi model, and cut the cost to roughly one tenth of the original:

~/.bashrc or ~/.zshrc 
export ANTHROPIC_BASE_URL="https://api.moonshot.cn/anthropic"
export ANTHROPIC_API_KEY="your Kimi API Key"

After that, run claude as usual. The framework is Claude Code, and the model is Kimi.

What Your Lab Can Use It For

The following five scenarios come directly from research workflows. Every one of them is something you can do today.

01 Batch data-file processing

You have 30 Excel files, each one containing measurement results from one sample, and the formats differ slightly.

"Help me merge these files, organize them by sample ID, output one summary table, and mark which samples have missing data."

It opens each file, handles the formatting differences, generates the result, and tells you when it notices anomalies along the way.

02 Figure generation

You need a figure that meets submission requirements, with specific formatting constraints.

"Use this dataset to draw a boxplot. The x-axis is treatment group, the y-axis is cell viability, color the control and experimental groups differently, save it as a 300 dpi tif, and also tell me the p-values for each group."

Previously, this meant either writing Python yourself or clicking through Origin step by step. Now you say one sentence and it runs the whole thing and tells you the result.

03 Reading papers to assist method design

You have a methods section from a reference paper that you need to evaluate and decide whether it fits your experiment.

"What statistical method did they use? I now have 3 groups with 6 samples in each. Is their method suitable for my setup? Are there better choices?"

It will not think through the scientific question for you, but it can quickly map the technical choices in the paper and save you the time of digging through background materials.

04 Journal submission formatting

Different journals have different requirements: Nature-series journals want double spacing and Vancouver references; some major Chinese journals require SimSun 5-point font and bilingual figure captions in Chinese and English; Cell-family journals use yet another system.

"Help me convert this manuscript into the submission format for Journal XXX. Follow their Author Guidelines, change Figure to Fig. throughout, and standardize the reference format."

If the same paper gets rejected and has to be submitted elsewhere, you can just say, "Reformat it according to Journal YYY" and you do not have to redo the whole thing by hand.

05 Taking over legacy code

A previous student left the lab and left behind a pile of uncommented scripts. The most important file is named analysis_final_v3_REAL_use_this.py. You do not know what it computes, where the data comes from, or what changes are safe.

"Read and understand this code. Tell me what the data flow is, what each script does, and what the naming pattern of the variables is."

It gives you a plain-language explanation and helps you understand the logic of the code. Then:

"We switched to a new instrument and the output format changed. Help me modify only the data-reading part and leave the rest of the logic untouched."

It can pinpoint exactly what needs to change without rewriting the whole script. When lab members leave, projects are handed off, or your advisor asks you to reproduce results from two years ago, AI effectively fills in as the always-online "senior person" your lab never documented.

What these tasks have in common is this: before, either you had to learn a language to do them, or you had to repeat the same manual operations again and again. Now the barrier becomes: you only need to know what result you want.

Going Further: Skills and the MCP Ecosystem

Skills: reusable expert experience

A Skill is a written instruction set that tells the AI how to handle a recurring type of task. It is like turning experience into a reusable template.

The important thing is: you do not have to write these Skills yourself. Claude Code has an open community where developers and researchers continuously contribute all kinds of Skills - statistical analysis, code review, literature reading, manuscript polishing, and more. You can install and use them directly.

Things a lab could write for itself:

/submission-format - Format a manuscript according to a journal's Author Guidelines
/statistical-test - Recommend an appropriate method based on the experimental design
/data-check - Automatically check for missing values, outliers, and inconsistent units
/abstract-polish - Check an abstract against a methods-results-conclusion structure and mark logical jumps

MCP: connect the tools you already use

MCP (Model Context Protocol) is an open protocol introduced by Anthropic that lets AI directly call external applications. There is a large open-source community ecosystem around it.

MCP What it can do
Notion / Feishu Automatically organize experiment records into a database and generate meeting notes
GitHub Read repositories, submit changes, and manage versions
Email client Draft reviewer responses and submission-confirmation emails
Literature databases Search PubMed and export citation formats, with additional setup

Official MCP list: modelcontextprotocol.io

Why the ecosystem matters

You do not need to be an expert right now, but you can directly use skills written by experts.

When a statistician packages an analysis method as a Skill, and when an experienced editor writes their polishing logic into a workflow, that experience becomes a capability you can call directly without learning it all from scratch.

First stand on the shoulders of giants and get the work done. In that process, you will gradually understand the logic underneath. Once you understand your lab's own workflow - how to clean this kind of data, what to watch out for when submitting to journals in this field, what your group's standard mass spectrometry workflow is - you can write that experience into a Skill and publish it so people around the world doing similar research can use it.

You are not just someone who uses tools. You can also be someone who adds capabilities to them.

Usage Tips

Distilled from Anthropic's internal practice and translated into a research context:

Principle Research interpretation
Demo first, not docs first Get AI to produce a figure first, then discuss whether the method is right. Five minutes to a result beats two pages of analysis planning.
Short-cycle experiments, not giant plans Do not try to "AI-ify the entire workflow." Start with the single most annoying task you have today.
Rerun old workflows with new models Something AI could not do three months ago may be possible now. Periodically retry old hard problems.
Do the simplest version first State what you want in the most direct language possible. Do not spend too much time tuning prompts up front.
Blur role boundaries You do not need to become a programmer to run code, and you do not need to become a statistician to do statistics. You need the eye to judge whether the result is right.

This Website Itself Is a Live Example

The page you are reading now was made using exactly the approach described in this article.

The process was this: I had the ideas and the judgment. I knew what a non-CS research group needed, which tools were worth recommending, and how the content logic should flow. The execution part - the design system, the HTML/CSS, the tool research, and the phrasing refinements - was handed to AI. My role was exactly the "approver" described in this article: look once, decide whether it works, request a change, and continue.

From initial idea to this page going live took only a few hours. If I had done it the traditional way - learning front-end development, hand-writing the code, and adjusting the styles line by line - the same result might have taken days or more.

This is not about showing how powerful AI is. It is about saying: the reason this works is the professional judgment you already have. AI has no way to know on its own what a research group cares about, which analogy will make the point clearest, or which details are worth emphasizing. Those all come from your accumulated understanding of the field.

How to do something similar with happycapy

The advantage of happycapy is simple: you open the browser and use it, with no environment setup at all. Here is a conversation starting point you can reuse directly:

Example From an Idea to a Page

"I want to make a beginner's guide for researchers in XX field, and the topic is YY. The readers have domain expertise but do not know programming. First help me sort out the content structure, then turn it into a simple HTML page that I can share."

Tell it your reference material, your core point, and who your readers are, all in natural language. It will help you break down the structure, draft the content, and generate the page. You are responsible for judging whether each step is correct and whether the direction is still right.

"The logic in section two is wrong. It should explain the problem before the solution. Help me adjust that, and leave the rest unchanged."

That is the whole thing. You do not need to learn HTML, understand CSS, or know what a server is. You only need to know what you want to say.
