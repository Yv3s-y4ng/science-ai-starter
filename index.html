实用入门指南

用 AI 做科研：非计算机课题组实用入门

把你已有的专业判断力，接上 AI 的执行力——不需要成为程序员。

16 个月。41 倍。 Anthropic 数据：AI 完成同一软件任务的速度，是 16 个月前的 41 倍。

Naval 说：软件被 AI 吃掉了。Anthropic 的产品负责人说：从想法到原型，已经从几周缩短到几小时。

我们正在进入一个执行力暴涨的时代。写代码、画图、出报告——这些"做出来"的门槛，正在以你我都没预料到的速度趋近于零。

但有一件事 AI 给不了你：判断力。

知道该做什么实验、知道这个数据哪里有问题、知道这篇稿子的逻辑在哪里断了——这来自你在这个领域里积累的专业直觉。有人说得很准：AI 的能力边界，就是你的知识边界、你掌握的专业词汇、你对这个领域细节的好奇心。

这才是每个人在 AI 时代真正的护城河。

这篇文档想讲的是：如何把你已有的专业判断力，接上 AI 的执行力。

你现在在用 AI 做什么

你可能在用 ChatGPT 改文字、翻译、解释 error message，或者生图，再复制粘贴回去。贴进去，得到回答，复制出来，再贴下一段。

这已经很有用了。但注意这个操作模式：所有的搬运都是你在做。你读文件、你复制内容、你跑代码、你拿回输出、你再贴进去。

这种模式有个名字：你在替 AI 打工。

你变成了 AI 和你真正工作之间的接口——一次又一次地手动搬运信息。AI 坐在玻璃后面；你是那个按所有按钮的人。

差在哪里

复制粘贴循环的存在，是因为今天大多数人使用 AI 的方式有四个结构性限制：

  • 它不认识你的文件。你得把数据复制进对话框——超过几十行就开始出错，或者直接把后面的内容忽略掉。
  • 它不能真正跑代码。它给你写了 Python，但你还得自己去装环境、跑、再把报错贴回来。
  • 每次新对话记忆清零。你的项目背景、上次的决定、数据格式——每次都要重新解释一遍。
  • 它不知道结果对不对。图生出来了，但 AI 自己不会看、不会判断结果是否合理。只有你能做这件事。

像一个记忆力为零、不会用电脑的高级顾问——你得把所有材料念给他听，他给意见，你回去自己执行，执行完你再来汇报，而他已经把上次的对话忘得干干净净。

这不是在批评 AI，而是在描述一种使用方式——一种可以改变的使用方式。

AI 终端智能体是什么

AI 终端智能体把玻璃拿掉了。它不再坐在对话框后面等你贴东西，而是直接操作你电脑上的文件和程序：打开文件、写代码、运行、看结果、发现问题、修改、再运行——全部自己来。

你说"帮我分析这份数据"——它会自己打开文件、写代码、运行、看结果、发现问题、修改、再运行，然后把结果给你看。中间你什么都没做。

你的角色从执行者变成了审批者。它干活，你决定结果要不要。

之前 复制粘贴循环

你打开文件。你复制一段数据。你贴进对话框。你问问题。你复制它返回的代码。你打开终端。你跑代码。你遇到报错。你复制报错。你贴回去。你复制修复方案。你再跑。你复制输出结果。你贴回去让它解读。你复制解读。你贴进你的笔记。

现在 终端智能体

你描述你想要什么。它打开文件、写代码、跑代码、读输出、发现报错、修复、重跑，把结果返回给你。你看一眼，决定要不要继续。

有一点需要说清楚：它会犯错。所以"审批者"这个角色不能省——这不是工作流的缺点，这就是工作流本身。判断力由你提供，执行力交给它。

Kimi 为什么靠谱

2026 年 3 月，一件有意思的事被曝光：Cursor（全球最受欢迎的 AI 编程工具之一）的新旗舰模型 Composer 2，被社区发现底层用的是 Moonshot AI（月之暗面）的 Kimi K2.5 模型。Cursor 联合创始人随后承认，"我们从 Kimi K2.5 的开源基础上继续训练"。Kimi 官方也回应："我们为 Kimi K2.5 成为 Composer 2 的基础感到骄傲。"

同一时期，另一家国内大模型公司 MiniMax 被指在自家产品中混入了 Kimi 的代码（连 kimi 字样都没改干净）——真实性存疑，但可以说明 Kimi 在代码能力上已经处于国内第一梯队。

省钱操作

在 Claude Code 里改两行环境变量，就能把底层模型换成 Kimi——框架用 Claude Code，模型用 Kimi，成本约降至原来的 1/10：

~/.bashrc 或 ~/.zshrc 
export ANTHROPIC_BASE_URL="https://api.moonshot.cn/anthropic"
export ANTHROPIC_API_KEY="你的 Kimi API Key"

之后照常运行 claude，框架是 Claude Code，模型是 Kimi。

课题组能用来做什么

以下五个场景直接来自科研工作流，每一个今天就能做。

01 批量处理数据文件

你有 30 个 Excel 表，每个是一个样品的测量结果，格式略有差异。

"帮我把这些文件合并，按样品编号整理，输出一张汇总表，标注哪些样品数据缺失。"

它会自己打开每个文件、处理格式不一致的问题、生成结果，中途发现异常还会告诉你。

02 生成图表

你需要符合投稿要求的图，有具体的格式规定。

"用这份数据画箱线图，x 轴是处理组别，y 轴是细胞活力，颜色区分对照和实验组，存成 300dpi tif，顺便告诉我各组的 p 值。"

之前这件事要么你自己写 Python，要么在 Origin 里一步一步操作。现在说一句话，它跑完告诉你结果。

03 读文献辅助方法设计

你有一篇参考文献的方法部分需要评估，看是否适用于你的实验。

"他们的统计方法是什么？我现在有 3 组、每组 6 个样本，用他们的方法合适吗？有没有更好的选择？"

它不会替你思考科学问题，但它能帮你快速摸清文献里的技术选型，节省你查资料的时间。

04 期刊投稿排版

不同期刊要求不一样：Nature 系列要双倍行距、参考文献 Vancouver 格式；某些中文核心要宋体五号、图题中英文双语；投 Cell 子刊又是另一套。

"帮我把这篇稿子改成《XXX》的投稿格式，参考他们官网的 Author Guidelines，图序从 Figure 改成 Fig.，参考文献格式统一。"

如果同一篇稿子被拒、要投另一个刊，再说一句："按《YYY》的格式重新排"——不用重新手工改一遍。

05 接手历史遗留代码

前任学生离组，留下一堆没有注释的脚本，最重要的文件叫 analysis_final_v3_REAL_use_this.py。你不确定它在算什么、数据从哪来、改哪里会出问题。

"读懂这些代码，告诉我数据流是什么、每个脚本在做什么、变量命名的规律是什么。"

它会给你一份白话说明，帮你搞清楚这段代码的逻辑。然后：

"我现在换了新的仪器，输出格式变了，帮我把数据读取部分改一下，其他逻辑不动。"

它能精准定位需要改的地方，不会把整个脚本翻一遍。组里换人、项目交接、导师让你复现两年前的结果——这些场景下，AI 等于帮你补了一个随时在线的"老人"。

这些任务的共同点——之前要么你学了语言才能做，要么反复手工操作。现在门槛变成：你知道自己想要什么结果就够了。

进一步扩展：Skill 与 MCP 生态

Skills：可复用的专家经验

Skill 是一段写好的指令集，告诉 AI 遇到某类任务该怎么做——相当于把经验固化成可重复调用的模板。

重要的是：这些 Skill 不需要你自己写。Claude Code 有一个开放社区，全球开发者和研究者持续贡献各类 Skill——统计分析、代码审查、文献阅读、论文润色……你可以直接安装使用。

课题组自己能写的：

/投稿格式 — 按期刊 Author Guidelines 排好稿子
/统计检验 — 根据实验设计推荐合适方法
/数据质检 — 自动检查缺失值、异常值、单位不一致
/摘要润色 — 按"方法-结果-结论"结构检查摘要，标出逻辑跳跃

MCP：连接你已经在用的工具

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，让 AI 能直接调用外部应用。有庞大的开源社区生态。

MCP 能做什么
Notion / 飞书 实验记录自动整理进数据库，生成会议纪要
GitHub 读仓库、提交修改、管理版本
邮件客户端 起草审稿回复、投稿确认邮件
文献数据库 查 PubMed、导出引用格式（需额外配置）

官方 MCP 列表：modelcontextprotocol.io

为什么生态很重要

你现在不需要是专家，但你可以直接用专家写好的技能。

当一个统计学家把他的分析方法封装成 Skill，当一个资深编辑把润色逻辑写进工作流，这些经验就变成了你可以直接调用的能力——不需要你从头学。

先站在巨人的肩膀上把事情做出来。在这个过程里，你会慢慢理解背后的逻辑。当你摸清了自己课题组的工作流——怎么清洗这类数据、投这个领域的期刊有哪些注意事项、你们组跑质谱的标准流程是什么——你就可以把这些经验写成 Skill，发出去，让全世界做同类研究的人都能用。

你不只是用工具的人，你也可以是给工具加能力的人。

使用提示

从 Anthropic 内部实践提炼，转译成科研语境：

原则 科研版解读
Demo 优先，而非文档优先 先让 AI 跑出一张图，再讨论方法对不对。与其写两页分析思路，不如五分钟出结果。
短周期试验，而非大计划 不要想着"把整个分析流程 AI 化"，先找今天最烦的一件事试试。
旧流程用新模型重跑 三个月前觉得 AI 做不到的事，现在可能可以了。定期把旧的难题再问一遍。
做最简单的版本 先用最直白的语言说清楚你要什么，别花太多时间调 prompt。
模糊角色边界 你不需要成为程序员才能跑代码，不需要是统计学家才能做统计——你需要的是判断结果对不对的眼光。

这个网站本身就是一个活案例

你现在看到的这个页面，是用本文描述的方式做出来的。

整个过程是这样的：我有想法和判断——知道非计算机课题组需要什么、知道哪些工具值得推荐、知道内容逻辑该怎么走。执行部分——设计系统、写 HTML/CSS、查工具信息、打磨措辞——全部交给 AI。我的角色就是本文说的那个「审批者」：看一眼，决定要不要，提出修改，继续。

从有想法到这个页面上线，几个小时内完成。如果用传统方式——自己学前端、手写代码、逐行调样式——同样的结果可能需要几天甚至更久。

这不是展示 AI 有多强。这是在说：你已有的专业判断力，才是这件事能成立的原因。 AI 没有办法凭空知道课题组在乎什么、哪个类比更容易被理解、哪些细节值得强调——这些都来自你对这个领域的积累。

如何用 happycapy 做类似的事

happycapy 的优势在于：打开浏览器就能用，不需要配置任何环境。下面是一个可以直接套用的对话起点：

示例 从一个想法到一个页面

「我想做一个面向 XX 领域研究者的入门指南，主题是 YY。读者是有专业背景但不懂编程的人。帮我先梳理一下内容结构，再用简单的 HTML 做成一个可以分享的网页。」

把你的参考资料、你想讲的核心观点、你的读者是谁——用自然语言告诉它。它会帮你拆解结构、起草内容、生成页面。你负责判断每一步的结果对不对、方向有没有偏。

「第二节的逻辑不对，应该先讲问题再讲解法。帮我调整一下，其他部分不动。」

这就是全部。不需要学 HTML，不需要懂 CSS，不需要知道什么是服务器。你只需要知道你想说什么。
